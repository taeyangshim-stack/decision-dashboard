# Repository Guidelines

## 프로젝트 구조 및 모듈 구성
- 애플리케이션 코드는 `src/`에 두고 기능별 폴더를 사용합니다(예: `src/features/decisions`, `src/components`, `src/lib`); 공유 유틸은 작고 타입 안전하게 유지합니다.
- 테스트는 `tests/`에서 `src/` 구조를 반영합니다(`tests/features/decisions/decision-form.test.tsx`); 픽스처는 `tests/fixtures/`에 둡니다.
- 정적 자산은 `public/`(파비콘, 폰트, mock JSON)으로, 문서는 필요 시 `docs/`로, 로컬 작업을 돕는 스크립트는 `scripts/`로 정리합니다.
- 모듈은 작고 조합 가능하게 유지하고, 교차 관심사는 `src/lib`로 분리해 순환 의존을 피합니다.

## 빌드, 테스트, 개발 커맨드
- 잠금 파일이 존재하면 동일한 패키지 관리자 명령을 사용합니다(`npm install`, `pnpm install`, `yarn install` 중 일관되게).
- 로컬 개발 서버: `npm run dev`(핫 리로드 포함 워치 모드).
- 프로덕션 번들 빌드: `npm run build`(타입 체크가 파이프라인에 묶여 있으면 타입 에러 시 실패).
- 자동화 테스트 실행: `npm test`(반복 작업 시 `--watch` 옵션 사용).
- 린트/포맷: 스크립트가 있으면 `npm run lint`, `npm run format`; 없으면 프로젝트 설정된 린터를 직접 실행 후 PR을 엽니다.

## 코딩 스타일 및 네이밍
- 들여쓰기는 2스페이스, 여러 줄 리터럴에는 트레일링 콤마 사용; 도구에서 강제하지 않는 한 홑따옴표를 선호합니다.
- 변수/함수는 camelCase, React 컴포넌트/클래스는 PascalCase, 파일·자산은 kebab-case를 사용합니다.
- 컴포넌트는 순수하게 유지하고 부작용은 훅/헬퍼로 분리합니다; 비직관적 결정에는 짧은 주석을 남깁니다.
- 서식과 import 정렬은 자동 포매터/린터에 따르며 수동 스타일 변경을 커밋하지 않습니다.

## 테스트 가이드
- 모듈명과 대응하는 단위·통합 테스트를 인접하게 둡니다(`*.test.ts` / `*.test.tsx`); 구현이 아닌 동작을 설명하는 이름을 씁니다.
- 버그 수정마다 회귀 테스트를 추가하고, 범용 변경이 아니라면 빠른 단위 테스트를 우선합니다.
- 대시보드 결정 로직과 데이터 변환 같은 핵심 경로를 의미 있게 커버합니다; UI 스냅샷은 결과가 안정적일 때만 사용합니다.

## 커밋 및 PR 가이드
- 커밋 메시지는 명령형으로 간결하게 작성합니다(예: “Add decision form validation”); 관련 변경을 묶어 검토 가능성을 높입니다.
- PR 전에는 빌드·테스트·린트를 실행하고, 요약/변경 이유/화면 영향을 포함한 스크린샷을 남깁니다.
- 관련 이슈를 연결하고, 마이그레이션·스키마 변경·새 환경 변수는 PR 설명에 명시합니다.
- PR 범위는 작게 유지하고, 무관한 정리는 후속 이슈로 분리하는 것을 선호합니다.

## 보안 및 설정 팁
- 비밀 키는 커밋하지 말고 `.env.local` 등 gitignore된 파일에서 불러옵니다.
- 대시보드와 결정 로직에 들어오는 입력은 검증·정규화하고, 네트워크 요청에는 명시적 에러 처리와 타임아웃을 둡니다.
- 새 의존성은 유지보수 중이며 라이선스가 명확한지 확인하고, 추가된 포트나 외부 서비스는 문서화합니다.
